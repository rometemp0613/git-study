## 오늘 배운 것

### cherry-pick - 특정 커밋만 가져오기

다른 브랜치의 **특정 커밋**을 현재 브랜치에 **복사**하는 명령어

```
main:     A --- B --- C
                 \
feature:          D --- E --- F

(main에서 git cherry-pick E)

main:     A --- B --- C --- E'    ← 새 해시!
                 \
feature:          D --- E --- F   ← 원본 그대로
```

### 핵심 명령어

| 명령어 | 용도 |
|--------|------|
| `git cherry-pick <해시>` | 특정 커밋을 현재 브랜치에 복사 |
| `git cherry-pick <해시1> <해시2>` | 여러 커밋 cherry-pick |
| `git cherry-pick --no-commit (-n)` | 커밋 안 하고 스테이징만 (합칠 때 유용) |
| `git cherry-pick --continue` | 충돌 해결 후 계속 (해시 붙이지 말 것!) |
| `git cherry-pick --abort` | cherry-pick 취소 |

### 핵심 포인트

- **복사**지 이동이 아님 → 원본 브랜치에 커밋 그대로 남아있음
- 복사된 커밋은 **새로운 해시** (부모 커밋이 달라서)
- `--continue`에 해시를 붙이면 에러남! merge/rebase도 마찬가지
- `--no-commit`으로 여러 커밋을 가져온 뒤 하나의 커밋으로 합칠 수 있음

### cherry-pick vs merge vs rebase

| 명령어 | 언제 쓸까 |
|--------|----------|
| cherry-pick | 특정 커밋 1~2개만 골라서 가져올 때 |
| merge | 브랜치 작업 끝나서 전체를 합칠 때 |
| rebase | 푸시 전 커밋 히스토리를 깔끔하게 정리할 때 |

## 실습 내용

1. 기본 cherry-pick - feature의 "D: 긴급 버그 수정" 커밋만 main에 가져오기
2. --no-commit - C, E 두 커밋을 하나의 커밋으로 합쳐서 가져오기
3. 충돌 만들기 - 양쪽에서 bugfix.txt 수정 후 cherry-pick → 충돌 해결 → --continue
4. --abort - cherry-pick 취소 명령어 확인

## 삽질 기록

- `echo "bugfix!"` → bash에서 `!`가 히스토리 확장 특수문자라 에러. `echo 'bugfix!'` (작은따옴표)로 해결
- `git cherry-pick --continue de0c65e` → 해시를 붙이면 안 됨! `git cherry-pick --continue`만 써야 함
