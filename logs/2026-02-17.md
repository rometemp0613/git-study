## 오늘 배운 것

### Pull Request - 코드 리뷰와 협업 워크플로우

PR은 feature 브랜치의 변경사항을 main에 합치기 전에 리뷰/검토하는 과정.

### PR 워크플로우

1. 브랜치 생성 (`git switch -c feature/xxx`)
2. 작업 후 커밋
3. push (`git push -u origin feature/xxx`)
4. PR 생성
5. 리뷰 & 수정
6. 머지
7. 브랜치 삭제

### 핵심 명령어 (GitHub CLI)

| 명령어 | 용도 |
|--------|------|
| `gh pr create --title "제목" --body "설명"` | CLI에서 PR 생성 |
| `gh pr view <번호>` | PR 정보 조회 |
| `gh pr merge <번호>` | PR 머지 |

### 3가지 머지 방법

| 방법 | 특징 | 언제 사용 |
|------|------|----------|
| Merge commit | 모든 커밋 보존 + 머지 커밋 생성 | 히스토리를 그대로 남기고 싶을 때 |
| Squash and merge | 모든 커밋을 1개로 합쳐서 머지 | 잡다한 커밋을 깔끔하게 정리할 때 |
| Rebase and merge | 머지 커밋 없이 일직선으로 붙임 | 커밋 수는 유지하되 깔끔한 히스토리 |
