# 해시 함수 — SHA-1과 커밋 ID의 원리

## 핵심 개념: 해시 함수란?

**입력 데이터를 고정 길이의 문자열로 변환하는 함수**. Git은 SHA-1 해시를 객체의 고유 식별자(키)로 사용한다.

```
입력 (아무 크기)          해시 함수          출력 (고정 크기)
─────────────────    ──────────────    ──────────────
"hello"              →  SHA-1  →      ce013625030ba8dba906f756967...
"hello world"        →  SHA-1  →      2aae6c35c94fcfb415dbe95f40...
(1GB 파일)           →  SHA-1  →      (똑같이 40자리)
```

---

## 해시 함수의 4가지 핵심 특성

| 특성 | 설명 | 예시 |
|------|------|------|
| **고정 길이 출력** | 입력이 1바이트든 1GB든 항상 같은 길이 | SHA-1 = 항상 40자리 16진수 |
| **단방향** | 해시값으로 원본 데이터를 역추적 불가 | `aaf4c6...` → `???` (불가능) |
| **결정적** | 같은 입력은 항상 같은 해시값 | `"hello"` → 항상 같은 해시 |
| **눈사태 효과** | 입력 1글자만 달라도 해시가 완전히 달라짐 | `"hello"` vs `"hellp"` → 완전히 다른 해시 |

## Git의 해시 계산 공식

Git은 모든 객체를 저장할 때 **헤더 + 내용**을 합쳐서 SHA-1을 계산한다.

```
SHA-1("{타입} {바이트수}\0{내용}")
       ^^^^^^ ^^^^^^^^^  ^^^^^^^
       객체 타입  크기    실제 데이터
                    \0 = null 바이트 (구분자)
```

### 객체별 공식

| 객체 | 공식 |
|------|------|
| blob | `SHA-1("blob {크기}\0{파일내용}")` |
| tree | `SHA-1("tree {크기}\0{tree 데이터}")` |
| commit | `SHA-1("commit {크기}\0{커밋 데이터}")` |

### 직접 검증하기

```bash
# Git이 계산한 해시
echo -n "hello" | git hash-object --stdin
# 출력: b6fc4c620b67d95f953a5c1c1230aaab5db5a1b0

# SHA-1 공식으로 직접 계산 (결과 동일!)
printf "blob 5\0hello" | sha1sum
# 출력: b6fc4c620b67d95f953a5c1c1230aaab5db5a1b0
```

## 해시 체인 — Git의 무결성 보장 원리

커밋 객체 안에는 **parent 커밋의 해시**가 포함되어 있다. 이것이 해시 체인을 형성한다.

```
commit C (hash: ccc...)
  ├── tree: ttt...
  ├── parent: bbb...  ←── B의 해시가 박혀있음
  └── message: ...

commit B (hash: bbb...)
  ├── tree: sss...
  ├── parent: aaa...  ←── A의 해시가 박혀있음
  └── message: ...

commit A (hash: aaa...)    ← root commit (parent 없음)
  ├── tree: rrr...
  └── message: ...
```

### 왜 조작이 불가능한가?

1. commit A의 내용을 수정하면 → A의 해시가 변함
2. commit B가 참조하는 parent 해시와 안 맞음 → B도 수정해야 함
3. B가 바뀌면 → C도 수정해야 함
4. **모든 후속 커밋이 연쇄적으로 바뀌어야 함 → 즉시 발각**

## .git/objects 저장 구조

해시값의 앞 2자리가 폴더명, 나머지 38자리가 파일명이 된다.

```
.git/objects/
├── 0e/d835c0267fce84b0f86104081f0d27d7d0cd1b   ← 커밋
├── aa/99e58f238c08f0d43e001ab0ba5d571f6511d4   ← tree
├── b6/fc4c620b67d95f953a5c1c1230aaab5db5a1b0   ← blob
└── ...
```

이렇게 나누는 이유: 한 폴더에 파일이 너무 많으면 파일시스템 성능이 떨어지기 때문.

## SHA-1 vs SHA-256

| | SHA-1 | SHA-256 |
|--|-------|---------|
| **출력 길이** | 160비트 (40자리 16진수) | 256비트 (64자리 16진수) |
| **보안** | 2017년 충돌 공격 성공 (SHAttered) | 현재 안전 |
| **Git 상태** | 기본값 (현재) | 실험적 지원 |
| **전환 명령어** | - | `git init --object-format=sha256` |

> 일반적인 Git 사용에서는 SHA-1 충돌 걱정할 필요 없음. 인위적 공격이 아닌 우연한 충돌 확률은 사실상 0.

## 명령어 정리표

| 명령어 | 설명 |
|--------|------|
| `echo -n "내용" \| git hash-object --stdin` | 내용의 SHA-1 해시 계산 (저장 안 함) |
| `git hash-object <파일>` | 파일 내용의 SHA-1 해시 계산 |
| `git log -1 --format="%H"` | 최근 커밋의 전체 해시 (40자리) |
| `git log -1 --format="%h"` | 최근 커밋의 축약 해시 (7자리) |
| `git cat-file -p <해시>` | 객체 내용 보기 (parent 해시 체인 확인용) |
| `printf "blob {크기}\0{내용}" \| sha1sum` | Git의 해시 공식을 직접 검증 |

---

## 주의사항 & 흔한 실수

- **16진수(hexadecimal)와 16비트를 혼동하지 말 것**: SHA-1은 16진수 40자리 = **160비트**
- **`git log --reverse -1`은 첫 커밋을 보여주지 않음**: `-1`이 먼저 적용됨. `git log --reverse`로 전체 보고 맨 위 확인
- **해시는 내용 기반**: 파일 이름이 달라도 내용이 같으면 같은 해시 (blob 특성과 연결)
